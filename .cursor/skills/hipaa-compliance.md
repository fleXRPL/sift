# HIPAA Compliance

Use this for US healthcare compliance. `healthcare-phi-compliance.md` is the primary implementation reference for PHI handling, data classification, audit logging, and leak prevention.

## When to Apply

- Any feature that stores, processes, exports, or transmits PHI
- Assessing whether logging, analytics, LLM prompts, or storage create HIPAA exposure
- Designing patient-facing or clinician-facing systems where minimum necessary access matters

## HIPAA Decision Gates

Before implementing any feature touching patient data, ask:

1. Is this data PHI?
2. Is access limited to the minimum necessary scope?
3. Are read/write/export events auditable?
4. Does any vendor or model provider require a BAA before touching the data?

## Non-Negotiable Guardrails

- Never place PHI in logs, analytics events, crash reports, LLM prompts, or client-visible error strings
- Never expose PHI in URLs, browser storage, screenshots, or example payloads
- Require authenticated access and audit trails for PHI reads and writes
- Treat third-party SaaS, observability tooling, and **LLM providers** as blocked-by-default until BAA status and data boundaries are clear
- Prefer opaque internal IDs over names, MRNs, phone numbers, addresses, or other direct identifiers

## Sift-Specific Notes

- LLM summaries generated by Sift are **derived PHI** — treat with the same sensitivity as source records
- The Ollama/local LLM runs on-device, which keeps PHI local. Never reconfigure Sift to send records to a remote/cloud LLM without a BAA review
- SQLite database file (`sift.db`) must not be stored in a world-readable location on the Windows machine
- The watch folder path may reveal patient context — do not log or expose it in error messages
